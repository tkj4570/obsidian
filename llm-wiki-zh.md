# LLM Wiki

一种使用 LLM 构建个人知识库的模式。

这是一份理念文档，设计用于复制粘贴到你自己的 LLM Agent 中（如 OpenAI Codex、Claude Code、OpenCode / Pi 等）。其目标是传达核心理念，但具体的实现细节将由你的 Agent 与你协作构建。

## 核心理念

大多数人对 LLM 和文档的体验就是 RAG：你上传一堆文件，LLM 在查询时检索相关片段，然后生成答案。这确实有用，但 LLM 每次回答问题都要从零开始重新发现知识。没有积累。问一个需要综合五份文档才能回答的微妙问题，LLM 每次都得重新找到并拼凑相关片段。什么都没被构建起来。NotebookLM、ChatGPT 文件上传以及大多数 RAG 系统都是这样工作的。

这里的思路不同。LLM 不是在查询时才从原始文档中检索，而是**增量构建并维护一个持久化的 wiki**——一个结构化、相互链接的 Markdown 文件集合，位于你和原始资料之间。当你添加新资料时，LLM 不只是为后续检索建立索引，而是阅读它、提取关键信息、将其整合到现有 wiki 中——更新实体页面、修订主题摘要、标注新数据与旧观点的矛盾之处、强化或挑战不断演进的综合论述。知识被编译一次，然后**持续保鲜**，而不是每次查询都重新推导。

这就是关键区别：**wiki 是一个持久化的、不断复利的产物。**交叉引用已经就位，矛盾已经被标记，综合论述已经反映了你读过的一切。每添加一个资料、每问一个问题，wiki 都变得更加丰富。

你几乎从不（或很少）自己写 wiki——LLM 编写和维护所有内容。==你负责策展来源、探索和提出正确的问题。LLM 做所有苦活——总结、交叉引用、归档和簿记，这些是让知识库长期有用的基础。==实践中，我一边开着 LLM Agent，一边开着 Obsidian。LLM 根据我们的对话进行编辑，我实时浏览结果——跟踪链接、查看图谱视图、阅读更新后的页面。Obsidian 是 IDE，LLM 是程序员，wiki 是代码库。

这可以应用于很多不同的场景。举几个例子：

- **个人**：追踪自己的目标、健康、心理、自我提升——归档日记条目、文章、播客笔记，随时间构建出关于自己的结构化图景。
- **研究**：在数周或数月内深入研究一个主题——阅读论文、文章、报告，增量构建一个带有演进论点的全面 wiki。
- **读书**：边读边归档每一章，为角色、主题、情节线索及其关联构建页面。读完时你就有了一个丰富的伴读 wiki。想想像 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 这样的粉丝 wiki——数千个相互链接的页面，涵盖角色、地点、事件、语言，由志愿者社区多年构建而成。你可以在阅读过程中个人构建类似的东西，由 LLM 完成所有交叉引用和维护工作。
- **商业/团队**：由 LLM 维护的内部 wiki，由 Slack 线程、会议记录、项目文档、客户通话提供养料。可能有人类参与审核更新。wiki 保持最新，因为 LLM 完成了团队中没人愿意做的维护工作。
- **竞品分析、尽职调查、旅行规划、课程笔记、兴趣深耕**——任何你需要随时间积累知识并希望它有条理而非散落各处的场景。

## 架构

有三个层次：

**原始资料**——你策展的源文件集合。文章、论文、图片、数据文件。这些是不可变的——LLM 从中读取但绝不修改。这是你的事实来源。

**Wiki**——LLM 生成的 Markdown 文件目录。摘要、实体页面、概念页面、对比、总览、综合论述。LLM 完全拥有这一层。它创建页面，在有新资料时更新它们，维护交叉引用，保持一切一致。你读它，LLM 写它。

**Schema**——==一份文档（如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md），告诉 LLM wiki 的结构、约定，以及在摄入资料、回答问题或维护 wiki 时应遵循的工作流。这是关键的配置文件——它让 LLM 成为一个有纪律的 wiki 维护者，而非一个通用聊天机器人。你和 LLM 随着摸索出适合你领域的做法，共同演进这份文档。==

## 操作

**Ingest（摄入）**。你把新资料放入原始资料集，然后告诉 LLM 去处理它。一个示例流程：LLM 读取资料，与你讨论关键要点，在 wiki 中写摘要页面，更新索引，更新整个 wiki 中相关的实体和概念页面，并向日志追加一条记录。一份资料可能触及 10-15 个 wiki 页面。我个人偏好一次摄入一份资料并保持参与——我阅读摘要、检查更新、引导 LLM 关注重点。但你也可以以较少监督的方式批量摄入多份资料。由你来摸索适合自己风格的工作流，并将其记录在 Schema 中供未来会话使用。

**Query（查询）**。你针对 wiki 提问。LLM 搜索相关页面，阅读它们，综合出带引用的答案。答案可以根据问题的不同采用不同的形式——Markdown 页面、对照表、幻灯片（Marp）、图表（matplotlib）、画布。重要的洞见：**好的答案可以作为新页面归档回 wiki。**你要求的对比、分析、发现的关联——这些都是有价值的，不应该消失在聊天记录中。这样你的探索就像摄入的资料一样在知识库中不断复利。

**Lint（检查）**。定期要求 LLM 对 wiki 进行健康检查。查找：页面之间的矛盾、被新资料取代的过时说法、没有入站链接的孤儿页面、被提及但缺少专属页面的重要概念、缺失的交叉引用、可以通过网络搜索填补的数据缺口。LLM 善于提出新的调查问题和新资料的建议。这让 wiki 在增长过程中保持健康。

## 索引与日志

两个特殊文件帮助 LLM（和你）在 wiki 增长时导航。它们服务于不同的目的：

**index.md** 是内容导向的。它是 wiki 中所有内容的目录——每个页面列出链接、一行摘要，可选地包含元数据如日期或资料数量。按类别组织（实体、概念、资料等）。LLM 在每次摄入后更新它。回答查询时，LLM 先读索引找到相关页面，然后深入它们。在中等规模（约 100 份资料、数百个页面）下这出奇地好用，避免了基于嵌入的 RAG 基础设施。

**log.md** 是按时间顺序的。它是一个只追加的记录，记录了什么发生了什么时间——摄入、查询、检查过程。一个有用技巧：如果每条记录以一致的格式开头（如 `## [2026-04-02] ingest | 文章标题`），日志就可以用简单的 Unix 工具解析——`grep "^## \[" log.md | tail -5` 显示最近 5 条记录。日志让你看到 wiki 演进的时间线，并帮助 LLM 理解最近做了什么。

## 可选：CLI 工具

到了某个阶段，你可能想构建一些小工具来帮助 LLM 更高效地操作 wiki。最显然的是 wiki 页面的搜索引擎——小规模时索引文件就够了，但随着 wiki 增长你会想要真正的搜索。[qmd](https://github.com/tobi/qmd) 是个不错的选择：它是 Markdown 文件的本地搜索引擎，具备混合 BM25/向量搜索和 LLM 重排序功能，全部在设备上运行。它既有 CLI（LLM 可以 shell out 调用），也有 MCP 服务器（LLM 可以当作原生工具使用）。你也可以自己构建更简单的东西——LLM 可以帮你在需要时 vibe-coding 一个简易搜索脚本。

## 技巧与小贴士

- **Obsidian Web Clipper** 是一个浏览器扩展，将网页文章转换为 Markdown。对于快速将资料收入原始资料集非常有用。
- **将图片下载到本地。**在 Obsidian 设置 → 文件和链接中，将"附件文件夹路径"设为一个固定目录（如 `raw/assets/`）。然后在设置 → 快捷键中，搜索"下载"找到"下载当前文件的附件"并绑定一个快捷键（如 Ctrl+Shift+D）。剪藏一篇文章后，按快捷键，所有图片就下载到本地磁盘了。这是可选的但很有用——它让 LLM 能直接查看和引用图片，而不是依赖可能失效的 URL。注意 LLM 不能原生地一次性读取带内联图片的 Markdown——变通方案是让 LLM 先读文字，然后单独查看部分或全部引用的图片以获取额外上下文。有点笨拙但效果不错。
- **Obsidian 的图谱视图**是查看 wiki 形态的最佳方式——什么连接什么、哪些页面是枢纽、哪些是孤儿。
- **Marp** 是基于 Markdown 的幻灯片格式。Obsidian 有相应的插件。用于直接从 wiki 内容生成演示文稿非常有用。
- **Dataview** 是一个 Obsidian 插件，在页面 Frontmatter 上运行查询。如果你的 LLM 给 wiki 页面添加 YAML Frontmatter（标签、日期、资料数量），Dataview 可以生成动态表格和列表。
- **Wiki 就是 Markdown 文件的 Git 仓库。**你免费获得版本历史、分支和协作能力。

## 为什么这行得通

维护知识库最繁琐的部分不是阅读或思考——而是簿记工作。更新交叉引用、保持摘要最新、标注新数据与旧观点的矛盾之处、在几十个页面间维护一致性。人们放弃 wiki 是因为维护负担增长得比价值更快。LLM 不会感到无聊、不会忘记更新交叉引用、可以一次触及 15 个文件。wiki 保持维护是因为维护成本几乎为零。

人类的职责是策展资料、指导分析、提出好问题、思考这一切意味着什么。LLM 的职责是其他所有事情。

这一理念在精神上与 Vannevar Bush 的 Memex（1945）相关——一个私人的、经过策展的知识存储，带有文档之间的联想路径。Bush 的愿景比后来互联网的形态更接近这个：私有的、主动策展的，文档之间的连接与文档本身同等珍贵。他没能解决的部分是谁来做维护。LLM 解决了这个问题。

## 附注

本文档刻意保持抽象。它描述的是理念，而非具体实现。确切的目录结构、Schema 约定、页面格式、工具——所有这些都取决于你的领域、你的偏好和你选择的 LLM。上面提到的一切都是可选和模块化的——选择有用的，忽略无用的。比如：你的资料可能只有纯文本，所以你完全不需要图片处理。你的 wiki 可能足够小，索引文件就是你需要的全部，不需要搜索引擎。你可能不在乎幻灯片，只想要 Markdown 页面。你可能想要一套完全不同的输出格式。正确的使用方式是把它分享给你的 LLM Agent，一起协作实例化出适合你需求的版本。这份文档的唯一职责是传达这个模式。你的 LLM 可以搞定其余的。
